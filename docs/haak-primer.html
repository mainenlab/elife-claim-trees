HAAK: Background for the Attribution Whitepaper
Zachary F. Mainen · Champalimaud Research

What HAAK Is

HAAK is an open-source platform for agentic research in which AI agents with defined roles, memory, and governance work alongside humans on long-horizon tasks. Agents are mortal: each session ends and the agent's context is destroyed. Continuity is maintained not through persistent model state but through externalisation — every decision, contribution, and state change is written to a shared repository before the session ends. A new agent inherits context from this record; an outgoing agent hands off to it. The institution persists even though the instances do not. This is the architectural motivation for the provenance ledger described below — it is built for continuity, not for attribution. Attribution is a consequence.

Four Contribution Types

In HAAK, value is created by four kinds of contributors, all treated as first-class entities in the same provenance record. Materials are data deposits: artworks, market records, behavioral signals, training datasets. Code is software contributions: algorithms, models, feature extractors, analysis pipelines. Policy is governance decisions: data-sharing agreements, constitutional amendments, access controls that determine which data becomes available to the system. Methods are protocol documents: written procedures that specify how agents perform tasks, how models are trained, how outputs are evaluated. The attribution whitepaper argues that Shapley values can be computed for all four types — not just materials — given the right infrastructure. That infrastructure is the ledger.

The Filix Ledger

Filix is HAAK's provenance ledger: a Merkle-CRDT structure that records every contribution to the system with cryptographic provenance, timestamps, and links to the model state before and after. Every data deposit, code commit, policy decision, and method update is a signed entry in the ledger. Every model version is a hash of the ledger state that produced it. Every performance measurement is anchored to a specific ledger state. This structure provides what Shapley computation requires: for any contribution c, the ability to identify all model versions that included c and compare their performance against versions that did not. The ledger makes the Monte Carlo approximation of Shapley values tractable across all four contribution types — not just data.

The Relational Ontology

HAAK's data model is relational in a specific sense: every entity exists through its relations to other entities, not through intrinsic properties. A contribution is not a row in a database — it is an entity that participates in typed relations (called belongings) with other entities: the model version it affected, the performance metric it changed, the policy that enabled its access, the method that governed its use. These typed relations are what allow Shapley attribution to span heterogeneous contribution types within a unified computational framework. When the whitepaper refers to contributions as "entities" and their relationships as "belongings," it is using HAAK's ontological vocabulary precisely: the Shapley computation traverses the belonging graph, not a flat list of training examples.
